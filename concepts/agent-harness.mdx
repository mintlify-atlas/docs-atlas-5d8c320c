An **agent harness** is the complete infrastructure that surrounds an LLM and turns it into a functional agent. The model supplies intelligence; the harness supplies everything else.

> The model is the agent. The code is the harness.

## The core idea

A large language model, on its own, can reason and generate text — but it cannot open a file, run a command, remember what happened last session, or ask you for permission before deleting something. Those capabilities belong to the harness layer.

The relationship is clean and deliberate:

| Layer | Responsibility |
|-------|---------------|
| **LLM** | Decide what to do next |
| **Harness** | Do it safely, efficiently, and with full observability |

This separation is not just architectural tidiness. It means you can swap the model (Anthropic → Kimi → any compatible endpoint) without rewriting your tools, skills, or permission rules. It also means you can audit every action the agent takes without touching the model itself.

## The harness equation

```
Harness = Tools + Knowledge + Observation + Action + Permissions
```

Each term maps to a concrete subsystem in OpenHarness:

## Why this separation matters

**Composability.** Each pillar is independently replaceable. You can swap in a custom permission backend, add new tools, or load a different skill set without touching the core loop.

**Safety.** The harness is the single enforcement point for every action the model wants to take. Permissions, hooks, and approval dialogs all live here — not scattered across individual tools.

**Observability.** Because all tool calls flow through a single pipeline (permission check → pre-hook → execute → post-hook), you get a complete audit trail of every agent action with no extra instrumentation.

**Portability.** Skills and plugins follow the same format as `anthropics/skills` and `claude-code/plugins`, so community-built knowledge and workflow extensions drop in without modification.

## How OpenHarness implements this pattern

OpenHarness is a Python implementation of the agent harness pattern. Its 14 subsystems map directly onto the five pillars above:

- The `engine` subsystem runs the core agent loop.
- The `tools` and `mcp` subsystems handle action.
- The `skills` and `prompts` subsystems handle knowledge and context.
- The `permissions` and `hooks` subsystems handle safety and observation.
- The `memory`, `tasks`, and `coordinator` subsystems handle persistence and coordination.
- The `plugins`, `commands`, `config`, and `ui` subsystems handle extensibility and user interaction.

See the [Architecture](/concepts/architecture) page for a full subsystem breakdown.
