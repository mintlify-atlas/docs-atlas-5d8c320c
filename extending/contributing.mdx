OpenHarness is a community-driven research project. Contributions of all kinds are welcome, from small bug fixes and documentation improvements to new tool implementations and provider support.

## Contribution areas

| Area | Examples |
|------|---------|
| **Tools** | New tool implementations for specific domains |
| **Skills** | Domain knowledge `.md` files (finance, science, DevOps, ...) |
| **Plugins** | Workflow plugins with commands, hooks, and agents |
| **Providers** | Support for more LLM backends (OpenAI, Ollama, ...) |
| **Multi-agent** | Coordination protocols and team patterns |
| **Testing** | E2E scenarios, edge cases, and benchmarks |
| **Documentation** | Architecture guides, tutorials, and translations |

## Development setup

<Steps>
  <Step title="Fork and clone the repository">
    ```bash
    git clone https://github.com/HKUDS/OpenHarness.git
    cd OpenHarness
    ```
  </Step>

  <Step title="Install dependencies">
    OpenHarness uses [`uv`](https://docs.astral.sh/uv/) for dependency management. The `--extra dev` flag installs test and lint tooling.

    ```bash
    uv sync --extra dev
    ```
  </Step>

  <Step title="Install frontend dependencies (optional)">
    Only required if you are working on the React terminal UI:

    ```bash
    cd frontend/terminal
    npm ci
    cd ../..
    ```
  </Step>

  <Step title="Verify the setup">
    Run the unit and integration test suite to confirm everything is working:

    ```bash
    uv run pytest -q
    ```

    You should see all 114 tests pass.
  </Step>
</Steps>

## Local checks

Run these checks before opening a pull request. They match what CI runs.

<CodeGroup>

```bash Lint
uv run ruff check src tests scripts
```

```bash Tests
uv run pytest -q
```

```bash Frontend type check
cd frontend/terminal && npx tsc --noEmit
```

</CodeGroup>

Type coverage is not yet a required green check for the whole repo, but you are welcome to run `uv run mypy src/openharness` if you are improving type annotations.

## Pull request expectations

- **Keep PRs scoped.** Small, focused changes merge faster than broad rewrites.
- **Describe the problem, the change, and how you verified it.** A one-paragraph summary covering all three is enough.
- **Add or update tests** when behavior changes. The `tests/` directory follows the same structure as `src/openharness/`.
- **Update docs** when CLI flags, workflows, or compatibility claims change.
- **Add a changelog entry.** Add a short line under `Unreleased` in [`CHANGELOG.md`](https://github.com/HKUDS/OpenHarness/blob/main/CHANGELOG.md) for any user-visible change.

## Running the full test suite

The unit and integration tests cover the core harness runtime. Additional E2E scripts exercise real model calls and plugin compatibility.

```bash
# Unit and integration tests (no model calls required)
uv run pytest -q

# Harness features E2E (requires API key)
python scripts/test_harness_features.py

# Real skills and plugins E2E (requires API key)
python scripts/test_real_skills_plugins.py
```

## Reporting bugs and requesting features

Use GitHub issues for bug reports and feature requests.

<CardGroup cols={2}>
  <Card title="Report a bug" icon="bug" href="https://github.com/HKUDS/OpenHarness/issues/new">
    Include your environment, the exact command you ran, and the full error output
  </Card>
  <Card title="Request a feature" icon="lightbulb" href="https://github.com/HKUDS/OpenHarness/issues/new">
    Describe the concrete workflow gap and the behavior you expect
  </Card>
</CardGroup>

When reporting a bug, include:
- Python version and OS
- The exact `oh` command you ran
- Full error output or unexpected behavior description

When proposing a feature, explain the workflow gap it fills. If the change is primarily documentation or maintenance, say so — it helps scope the work as a docs PR.
